# Requirements Document

## Introduction

The Investment & Expense Tracker is a personal finance management application that uses Google Sheets as its database, a Google Apps Script web application as its API, and a React 19 + Vite Progressive Web App (PWA) as its mobile interface. It lets a single authorized user (with optional family allowlist) track daily expenses, SIP mutual fund investments, and stock holdings, and review consolidated monthly and net-worth summaries — all at zero operational cost and with complete data ownership inside the user's own Google account.

This document reverse-engineers the **existing** implementation across the full stack: the Apps Script backend (`01_Main.gs` through `08_WebAppAPI.gs`), the React PWA (`pwa/src/`), and the Google Sheets data model. It describes the system as currently built. It is the basis for subsequent verification-and-alignment tasks, not for net-new feature work. Items listed as "Future Considerations" in the source `.spec/` documents are explicitly out of scope.

## Glossary

- **Tracker_System**: The complete Investment & Expense Tracker application, spanning the PWA, the Apps Script API, and the Google Sheets database.
- **PWA_Client**: The React 19 + Vite Progressive Web App front end hosted on GitHub Pages.
- **API_Service**: The Google Apps Script web application exposing `doGet` and `doPost` endpoints (`08_WebAppAPI.gs`).
- **Sheets_Database**: The Google Sheets workbook acting as the persistent data store, containing the Daily Expenses, SIP Tracker, Stock Portfolio, Monthly Summary, and Daily Dashboard tabs.
- **Service_Worker**: The PWA service worker generated by `vite-plugin-pwa`, responsible for app-shell precaching and runtime API caching.
- **Offline_Queue**: The `localStorage` store (key `offlineExpenseQueue`) that holds expenses entered while the device is offline.
- **Auth_Module**: The authentication and authorization mechanism using Google Sign-In (OAuth 2.0), client-side email allowlist checks, and server-side token verification against the API_Service email allowlist.
- **Expense_Record**: A single daily expense entry (date, category, description, payment mode, amount, notes, auto-derived month-year).
- **SIP_Record**: A single systematic investment plan entry for a mutual fund.
- **Stock_Record**: A single stock holding entry in the portfolio.
- **GOOGLEFINANCE**: The Google Sheets built-in function used to auto-populate current market price and day-change values for stocks.
- **Authorized_User**: A user whose Google account email is present in both the PWA_Client allowlist and the API_Service allowlist.

## Requirements

### Requirement 1: Daily Expense Tracking

**User Story:** As an Authorized_User, I want to record daily expenses with category, amount, and payment details, so that I can track my spending against my budget.

#### Acceptance Criteria

1. WHEN an Authorized_User submits an expense with an amount and a selected category, THE Tracker_System SHALL persist an Expense_Record containing date, category, description, payment mode, amount, and notes to the Daily Expenses tab of the Sheets_Database.
2. WHERE an expense date is not supplied, THE PWA_Client SHALL default the expense date to the current date.
3. THE PWA_Client SHALL offer expense categories limited to Food, Transport, Shopping, Bills, Entertainment, Health, Education, Subscriptions, Groceries, Rent, EMI, Gifts, and Miscellaneous.
4. THE PWA_Client SHALL offer payment modes limited to UPI, Cash, Credit Card, Debit Card, Net Banking, and Wallet.
5. THE PWA_Client SHALL present quick-amount preset buttons for ₹50, ₹100, ₹200, ₹500, ₹1000, and ₹2000.
6. WHILE either the amount is empty or no category is selected, THE PWA_Client SHALL keep the expense submit control disabled.
7. WHEN an Expense_Record is written to the Daily Expenses tab, THE Sheets_Database SHALL auto-populate a Month-Year helper value derived from the expense date.

### Requirement 2: SIP (Mutual Fund) Tracking

**User Story:** As an Authorized_User, I want to record and monitor my SIP mutual fund investments, so that I can see my accumulated value and returns.

#### Acceptance Criteria

1. WHEN an Authorized_User submits a SIP entry, THE Tracker_System SHALL persist a SIP_Record containing fund name, AMC, monthly amount, SIP date, start date, and total months to the SIP Tracker tab of the Sheets_Database.
2. THE PWA_Client SHALL restrict the SIP date input to values from 1 to 28.
3. WHEN SIP units and current NAV are present for a SIP_Record, THE Sheets_Database SHALL compute Current Value as units multiplied by NAV.
4. WHEN monthly amount and total months are present for a SIP_Record, THE Sheets_Database SHALL compute Total Invested as monthly amount multiplied by total months, Returns as Current Value minus Total Invested, and Return Percentage as Returns divided by Total Invested multiplied by 100.
5. THE Sheets_Database SHALL present an aggregate total row summarizing all active SIP_Records.

### Requirement 3: Stock Portfolio Tracking

**User Story:** As an Authorized_User, I want to record my stock holdings and see live valuations, so that I can monitor portfolio profit and loss.

#### Acceptance Criteria

1. WHEN an Authorized_User submits a stock entry, THE Tracker_System SHALL persist a Stock_Record containing stock name, NSE symbol, sector, buy date, quantity, average buy price, and notes to the Stock Portfolio tab of the Sheets_Database.
2. WHEN an Authorized_User types a stock symbol in the PWA_Client, THE PWA_Client SHALL convert the symbol input to uppercase.
3. THE PWA_Client SHALL restrict the stock sector to IT, Banking, FMCG, Pharma, Auto, Energy, Metals, Infra, Realty, Telecom, Chemical, or Other.
4. WHEN quantity and average buy price are present for a Stock_Record, THE Sheets_Database SHALL compute Invested as quantity multiplied by average buy price.
5. THE Sheets_Database SHALL populate current market price and day-change percentage for each Stock_Record using GOOGLEFINANCE keyed on the NSE symbol.
6. WHEN current market price is available for a Stock_Record, THE Sheets_Database SHALL compute Current Value as quantity multiplied by current market price, Profit-and-Loss as Current Value minus Invested, and Profit-and-Loss Percentage as Profit-and-Loss divided by Invested multiplied by 100.
7. THE Sheets_Database SHALL present a portfolio total row summarizing aggregate invested, current value, and profit-and-loss.

### Requirement 4: Monthly Summary Dashboard

**User Story:** As an Authorized_User, I want a consolidated monthly and net-worth view, so that I can understand my savings rate and asset allocation.

#### Acceptance Criteria

1. THE Sheets_Database SHALL present a 12-month summary with columns for month, income, total expenses, SIP investment, stock buy, total investment, net savings, and savings rate percentage.
2. WHEN daily expenses exist for a given month, THE Sheets_Database SHALL auto-populate that month's total expenses in the Monthly Summary tab from the Daily Expenses tab.
3. THE Sheets_Database SHALL present a net-worth snapshot covering SIP/mutual funds, stocks, bank balance, fixed deposits, and other assets, with a total net worth value and a percentage allocation per asset type.
4. THE Sheets_Database SHALL present a current-month expense breakdown by category, including each category's percentage of the month total.

### Requirement 5: Budget Management

**User Story:** As an Authorized_User, I want configurable per-category budgets with alerts, so that I know when my spending approaches or exceeds a limit.

#### Acceptance Criteria

1. THE Tracker_System SHALL support a configurable monthly budget per expense category with default values that an Authorized_User can override.
2. WHILE a category's month-to-date spend is below 80 percent of its budget, THE Tracker_System SHALL indicate that category's status as OK.
3. WHILE a category's month-to-date spend is from 80 percent up to 100 percent of its budget, THE Tracker_System SHALL indicate that category's status as Warning.
4. WHILE a category's month-to-date spend exceeds 100 percent of its budget, THE Tracker_System SHALL indicate that category's status as Over Budget.
5. WHEN the budget check runs, THE Tracker_System SHALL produce an alert listing over-budget and warning categories.

### Requirement 6: Google Sheets Integration

**User Story:** As an Authorized_User, I want the workbook structured and formatted automatically, so that my data stays consistent and readable.

#### Acceptance Criteria

1. THE Sheets_Database SHALL maintain the Daily Expenses, SIP Tracker, Stock Portfolio, Monthly Summary, and Daily Dashboard tabs.
2. THE Sheets_Database SHALL enforce data-validation dropdowns on category and status columns.
3. THE Sheets_Database SHALL apply conditional formatting that displays profit values in green and loss values in red.
4. THE Sheets_Database SHALL freeze header rows on data tabs.
5. WHEN a row is edited on a data tab, THE Sheets_Database SHALL auto-fill dependent formulas via the onEdit trigger.
6. THE Sheets_Database SHALL provide a custom "Finance Tracker" menu with quick-add actions and HTML quick-add dialogs for expenses and stocks.

### Requirement 7: Utility Functions

**User Story:** As an Authorized_User, I want data-quality and reporting helpers, so that I can review and export my records.

#### Acceptance Criteria

1. WHEN the duplicate-detection utility runs, THE Tracker_System SHALL highlight Expense_Records that share the same date, category, and amount.
2. WHEN an Authorized_User requests an export, THE Tracker_System SHALL produce the current month's Expense_Records in CSV form viewable within a dialog.
3. WHEN an Authorized_User requests a monthly report, THE Tracker_System SHALL present category-wise spending for the current month in an alert dialog.

### Requirement 8: PWA Installation and App Shell

**User Story:** As an Authorized_User, I want to install the app on my phone and use it like a native app, so that I have quick, chrome-free access.

#### Acceptance Criteria

1. THE PWA_Client SHALL be installable to the device home screen on iOS Safari and on Android Chrome.
2. THE PWA_Client SHALL declare a web app manifest with name "Investment Tracker", short name "Tracker", 192x192 and 512x512 icons, standalone display mode, and portrait orientation.
3. THE PWA_Client SHALL declare a manifest theme color of #1a73e8, background color of #ffffff, scope of /InvestmentTracker/, and start URL of /InvestmentTracker/.
4. THE Service_Worker SHALL precache the app-shell files, including HTML, CSS, JavaScript, and icons.
5. WHEN a new deployment is available, THE Service_Worker SHALL auto-update using the autoUpdate registration type.

### Requirement 9: Mobile UI and Navigation

**User Story:** As an Authorized_User, I want a mobile-first interface with clear navigation and feedback, so that I can complete tasks quickly on a phone.

#### Acceptance Criteria

1. THE PWA_Client SHALL present a fixed bottom tab bar with Expense, Stock/SIP, and Dashboard tabs and SHALL highlight the active tab.
2. THE PWA_Client SHALL apply iOS safe-area insets so that navigation and content clear the device notch and home indicator.
3. THE PWA_Client SHALL provide a Stock/SIP screen that switches between a stock form and a SIP form.
4. THE PWA_Client SHALL provide a Dashboard screen with Today, Stocks, and SIPs sub-views and a refresh control that reloads data.
5. THE PWA_Client SHALL display profit values in green and loss values in red on the Dashboard.
6. WHEN an action succeeds or fails, THE PWA_Client SHALL display a toast notification, styled green for success and red for error, that auto-dismisses after 3 seconds.

### Requirement 10: Authentication and Authorization

**User Story:** As an Authorized_User, I want secure sign-in that only permits allowed accounts, so that my financial data stays private.

#### Acceptance Criteria

1. THE Auth_Module SHALL authenticate users through Google Sign-In using OAuth 2.0 without storing any passwords.
2. WHEN an Authorized_User signs in, THE PWA_Client SHALL verify the account email against its allowlist before issuing any API request.
3. WHEN the API_Service receives a request, THE API_Service SHALL verify the supplied token with Google and confirm the account email is present in its allowlist before processing the request.
4. IF the account email is not present in the API_Service allowlist, THEN THE API_Service SHALL return an error and THE PWA_Client SHALL clear the session and return the user to sign-in.
5. WHEN a stored token has expired, THE PWA_Client SHALL prompt the user to re-authenticate.

### Requirement 11: API Read and Write Operations

**User Story:** As an Authorized_User, I want the app to read and write my data reliably through the API, so that entries and dashboards stay in sync with the Sheets_Database.

#### Acceptance Criteria

1. WHEN the PWA_Client submits an expense, stock, or SIP, THE API_Service SHALL accept the request via POST and SHALL support a GET fallback for cross-origin handling.
2. WHEN the PWA_Client requests a summary, THE API_Service SHALL return today's total, month total, category breakdown, and today's entries.
3. WHEN the PWA_Client requests holdings, THE API_Service SHALL return all Stock_Records with current values and all SIP_Records with returns.
4. WHEN the PWA_Client requests reference data, THE API_Service SHALL return the category and payment-mode lists.
5. WHEN the PWA_Client submits a bulk expense sync, THE API_Service SHALL accept multiple Expense_Records in a single request.

### Requirement 12: API Error Handling

**User Story:** As an Authorized_User, I want clear handling of failures, so that I understand what happened and can recover.

#### Acceptance Criteria

1. IF an API request fails due to a network error while adding an expense, THEN THE PWA_Client SHALL add the Expense_Record to the Offline_Queue.
2. IF the API_Service returns an unauthorized response, THEN THE PWA_Client SHALL clear the session and return the user to sign-in.
3. IF the API_Service returns an invalid response, THEN THE PWA_Client SHALL display an error toast.
4. IF an API request times out, THEN THE PWA_Client SHALL present a retry option.

### Requirement 13: Offline Queue and Sync

**User Story:** As an Authorized_User, I want to add expenses without a connection and have them sync later, so that I never lose an entry.

#### Acceptance Criteria

1. WHILE the device is offline, THE PWA_Client SHALL store submitted Expense_Records in the Offline_Queue under the key `offlineExpenseQueue`, each with a timestamp.
2. WHILE the Offline_Queue is non-empty, THE PWA_Client SHALL display the queued-item count as a badge and SHALL show an offline indicator when no network is available.
3. WHEN the device transitions to online, THE PWA_Client SHALL send all queued Expense_Records to the API_Service in a single bulk request and SHALL clear the Offline_Queue upon success.
4. WHILE the device is offline, THE PWA_Client SHALL serve the app shell and SHALL display the most recent cached API data on the Dashboard.
5. THE Service_Worker SHALL cache API responses using a network-first strategy with a 24-hour expiry.
